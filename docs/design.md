# Часть 1. Проектирование и анализ

## 1.1. Пользовательские сценарии

1. Как игрок, я хочу выбрать тематическую подборку и пройти короткую сессию на 10 вопросов, чтобы проверить свои знания в конкретной категории.
2. Как игрок, я хочу кликнуть по карте и сразу получить расстояние, очки и правильную точку, чтобы быстро понимать, насколько точным был мой ответ.
3. Как игрок, я хочу после завершения игры увидеть лидерборд, свой итоговый рейтинг и персональный паспорт экспедиции, чтобы понимать собственный стиль игры и возвращаться в викторину снова.

## 1.2. Функциональные требования

### Обязательные

#### Frontend

- форма ввода имени игрока и запуск новой игры;
- отображение текущего вопроса, подсказки, прогресса и суммарного счёта;
- показ подсказки только по кнопке, а не автоматически;
- интерактивная карта мира с обработкой клика;
- отправка ответа без перезагрузки страницы;
- отображение результата ответа: расстояние, очки, правильная точка и факт о месте;
- переход к следующему вопросу внутри SPA;
- экран завершения игры с итоговым результатом, счётом и местом игрока в рейтинге;
- отображение таблицы лидеров.

#### Backend

- хранение и выдача активных вопросов;
- создание игровой сессии и очереди вопросов;
- фильтрация вопросов по подборке;
- проверка ответа игрока;
- расчёт расстояния и очков;
- сохранение ответов и итогов игры;
- выдача таблицы лидеров;
- валидация входных данных и обработка ошибок.

### Опциональные, реализованные в проекте

- тематические подборки: `Глобальный микс`, `Иконические места`, `Чудеса света`, `Столицы и города`;
- форма, через которую пользователь может предложить свой вопрос;
- фото объекта в карточке вопроса;
- англоязычный картографический слой;
- продуктовая killer feature: `Паспорт экспедиции` с титулом игрока, средней ошибкой, лучшим попаданием, числом открытых подсказок и рекомендацией следующей экспедиции.

### Опциональные, оставленные как идеи развития

- бесконечный режим;
- авторизация и персональный профиль;
- модерация пользовательских вопросов через административную панель;
- таймер с отдельным рейтингом на скорость.

## 1.3. Проектирование API

Бэкенд использует REST API. Авторизация в MVP не требуется.

### `GET /api/health`

Проверка доступности сервиса.

Ответ:

```json
{
  "status": "ok",
  "repository": "memory",
  "timestamp": "2026-03-19T12:00:00.000Z"
}
```

### `GET /api/themes`

Возвращает доступные подборки с описанием и количеством активных вопросов.

Ответ:

```json
{
  "items": [
    {
      "id": "all",
      "label": "Глобальный микс",
      "description": "Разные типы вопросов подряд: города, знаковые места и чудеса света.",
      "questionCount": 37
    }
  ]
}
```

### `POST /api/sessions`

Создание новой игровой сессии.

Тело запроса:

```json
{
  "playerName": "Alex",
  "themeId": "world-wonders"
}
```

Ответ:

```json
{
  "session": {
    "id": "session-id",
    "playerName": "Alex",
    "mode": "classic",
    "themeId": "world-wonders",
    "currentIndex": 0,
    "totalQuestions": 10,
    "totalScore": 0,
    "status": "active"
  },
  "question": {
    "id": "question-id",
    "prompt": "Где находится Тадж-Махал?",
    "hint": "Он находится в Индии.",
    "type": "landmark",
    "region": "Asia",
    "difficulty": "medium",
    "theme": "world-wonders",
    "imageUrl": null
  }
}
```

### `GET /api/sessions/:sessionId/current`

Возвращает текущее состояние игровой сессии и следующий активный вопрос.

### `POST /api/answers`

Проверка ответа игрока.

Тело запроса:

    "mode": "classic",
    "themeId": "world-wonders",
    "currentIndex": 1,

### `GET /api/leaderboard?limit=5&currentSessionId=session-id`

Возвращает лучших игроков по итоговым очкам. Если передан `currentSessionId`, API дополнительно возвращает место текущей сессии в общем рейтинге.

Ответ:

```json
{
  "items": [
    {
      "sessionId": "session-id",
      "playerName": "Alex",
      "totalScore": 6240,
      "answeredQuestions": 10,
      "finishedAt": "2026-03-19T12:30:00.000Z"
    }
  ],
  "currentRank": 3,
  "totalFinishedSessions": 14
}
```

### `POST /api/question-suggestions`

Сохраняет пользовательское предложение нового вопроса.

Тело запроса:

```json
{
  "playerName": "Alex",
  "prompt": "Где находится Колизей?",
  "locationName": "Колизей",
  "themeId": "iconic-landmarks",
  "notes": "Хорошо дополняет европейскую подборку",
  "sourceUrl": "https://ru.wikipedia.org/wiki/Колизей"
}
```

Ответ:

```json
{
  "message": "Спасибо! Мы сохранили предложение и сможем добавить его после проверки."
}
```

## 1.4. Модель данных

### `questions`

| Поле | Тип | Описание |
| --- | --- | --- |

| `prompt` | `text` | Текст вопроса |
| `hint` | `text` | Подсказка |
| `type` | `text` | `landmark / city / country` |
| `latitude` | `double precision` | Правильная широта |
| `longitude` | `double precision` | Правильная долгота |
| `region` | `text` | Географический регион |
| `difficulty` | `text` | Сложность |
| `theme` | `text` | Тематическая подборка |
| `fact` | `text` | Факт для экрана ответа |
| `image_url` | `text` | Опциональная ссылка на изображение |
| `is_active` | `boolean` | Доступность вопроса |

### `game_sessions`

| Поле | Тип | Описание |
| --- | --- | --- |
| `id` | `uuid` | Идентификатор игровой сессии |
| `player_name` | `text` | Имя игрока |
| `mode` | `text` | Игровой режим |
| `theme_id` | `text` | Выбранная подборка |
| `question_queue` | `jsonb` | Очередь ID вопросов на сессию |
| `current_index` | `integer` | Сколько вопросов уже отвечено |
| `total_questions` | `integer` | Размер сессии |

### `answers`

| Поле | Тип | Описание |
| --- | --- | --- |

## 1.5. Ключевые технические решения

- `React + TypeScript + Vite` на фронтенде: быстрый запуск, удобная компонентная модель и простой handoff для дальнейшей доработки интерфейса.
- `Leaflet + React-Leaflet`: лёгкая библиотека для карты, хорошо подходит для сценария с кликом по миру и визуализацией правильной точки.
- `Fastify + TypeScript` на бэкенде: быстрый REST API с чистой структурой и низкой инфраструктурной сложностью.
- `Supabase Postgres`: управляемая PostgreSQL-база для вопросов, игровых сессий, ответов и пользовательских предложений.
- `Vitest + GitHub Actions`: автоматические unit- и интеграционные backend-тесты, а также CI-проверка сборки и тестов на каждый `push` и `pull_request`.
- Архитектура выбрана как модульный монолит, разделённый на `frontend` и `backend`. Для учебного проекта это рациональнее микросервисов: проще запуск, тестирование, отладка и проверка.
- В проекте есть `in-memory` режим backend для локальной демонстрации без БД, а также полноценный режим с `Supabase`.
- Для карты используется англоязычный слой `CARTO`, чтобы подписи были единообразны и не переключались между разными локалями.
- Для фото применяется fallback через Wikipedia REST API: это даёт визуальный контекст без отдельного медиа-бэкенда.
- Подсказки сделаны скрытыми по умолчанию и открываются только по запросу игрока, чтобы интерфейс не спойлерил ответ слишком рано.
- Продуктовое улучшение проекта: после завершения сессии игрок получает `Паспорт экспедиции` с персональным титулом, точностью, статистикой подсказок и рекомендацией следующей подборки. Это повышает реиграбельность и превращает разовую викторину в цепочку коротких “экспедиций”.
