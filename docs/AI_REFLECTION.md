# Часть 3. Работа с AI

## 3.1. Применённые инструменты и промпты

В ходе выполнения задания использовались AI-инструменты:

- `Codex / ChatGPT` — проектирование архитектуры, генерация каркаса fullstack-приложения, подготовка документации, доработка продуктовых фич.

Примеры промптов:

1. `Предложи стек и архитектуру для географической викторины с картой и подсчётом очков. БД хочу на Supabase.`
   Использовался для выбора общего подхода: frontend, backend, формат API и модель хранения данных.

2. `Спроектируй REST API для викторины: старт сессии, получение вопроса, отправка ответа, лидерборд.`
   Использовался для подготовки backend-структуры и документации в `design.md`.

3. `Сформируй SQL-схему для Supabase Postgres под questions, game_sessions, answers и пользовательские предложения.`
   Использовался для вынесения структуры БД в отдельные SQL-файлы.

4. `Придумай продуктовые улучшения для географической викторины, которые реально повышают вовлечённость, а не просто добавляют декор.`
   Использовался для выбора тематических подборок, формы предложений и killer feature с паспортом экспедиции и рекомендацией следующей сессии.

5. `Расширь пул географических вопросов так, чтобы каждая тематическая подборка ощущалась полноценным режимом, а подсказки не раскрывали ответ слишком рано.`
   Использовался для доработки контентной части проекта и улучшения UX во время игры.


## 3.2. Анализ эффективности

AI оказался особенно полезен на этапах:

- выбора подходящего стека без избыточной сложности;
- быстрого проектирования API и модели данных;
- генерации базового каркаса приложения и документации;
- поиска продуктовых идей, которые можно встроить в MVP без перегрузки проекта.

Где AI помог ускорить разработку:

- быстро предложил архитектуру `React + Fastify + Supabase` вместо перегруженного варианта с микросервисами;
- помог разложить функциональность по сущностям, endpoint’ам и таблицам;
- сократил время на подготовку `README`, `design.md`, SQL-схемы и API-контрактов;
- ускорил доработку необязательных функций из ТЗ: лидерборда, тематических подборок, формы для пользовательских предложений и финального паспорта экспедиции;
- помог быстро добавить автоматические тесты и CI-проверки без отдельного ручного проектирования инфраструктуры с нуля.

Где AI требовал проверки и корректировки:

- часть предложений по структуре проекта пришлось адаптировать под реальное ТЗ и учебный формат;
- некоторые автоматически сгенерированные решения нужно было упрощать, чтобы приложение оставалось понятным и легко запускалось на Windows и Linux;
- продуктовые идеи приходилось фильтровать: не все “эффектные” предложения были полезны для конкретного учебного проекта.

Как обнаруживались ошибки:

- через сверку с исходным ТЗ;
- через проверку того, что каждая заявленная функция действительно поддерживается API и моделью данных;
- через локальный прогон, сборку Docker-конфигурации и прогон автотестов.

Были ситуации, когда от предложений AI пришлось отказаться:

- от идеи микросервисной архитектуры, потому что для учебного fullstack-проекта это создаёт лишнюю сложность;
- от слишком тяжёлых backend-фреймворков, которые не давали заметной пользы для MVP;
- от избыточно декоративных UI-идей, которые ухудшали бы читаемость карты и игрового интерфейса.

## 3.3. Выводы и рекомендации

1. AI особенно полезен как ускоритель рутинных этапов: проектирование каркаса, API, базы данных и документации.
2. AI не заменяет инженерное решение: его предложения нужно сверять с ТЗ, ограничениями проекта и здравым смыслом.
3. Наилучший эффект достигается, когда AI используется итеративно: сначала для черновика, затем для доработки и уточнения конкретных частей.

В будущих проектах AI стоит использовать для:

- быстрого старта новых сервисов;
- генерации boilerplate-кода;
- подготовки документации и технических заметок;
- обсуждения альтернативных архитектурных и продуктовых решений.

Коллегам, начинающим использовать AI в разработке, можно посоветовать:

- задавать узкие и конкретные запросы;
- всегда проверять код и архитектурные решения вручную;
- использовать AI не как источник окончательной истины, а как инструмент ускорения и поиска вариантов.
