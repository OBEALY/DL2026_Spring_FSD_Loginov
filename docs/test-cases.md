# Тест-кейсы

Документ описывает реальные проверки проекта `Geo-quiz`: автоматические тесты backend, CI-проверки на GitHub и ручные smoke-сценарии для полного приложения.

## 1. Область проверки

Покрываются:

- доступность frontend и backend;
- API для старта игры, получения подборок, отправки ответов и получения лидерборда;
- полный игровой цикл на 10 вопросов;
- форма предложения пользовательского вопроса;
- расчёт расстояния, scoring и accuracy labels;
- сборка проекта;
- автоматический прогон на `push` и `pull_request`.

## 2. Среда выполнения

Основной сценарий для проверяющего:

- запуск: `Docker Compose`
- режим данных: `in-memory`
- frontend: `http://localhost:5173`
- backend: `http://localhost:4000`

Автоматические тесты:

- backend: `Vitest`
- CI: `GitHub Actions`
- репозиторий для CI: `main` branch на GitHub

Дополнительный сценарий:

- backend может работать и через `Supabase`, если указать `SUPABASE_URL`, `SUPABASE_SERVER_KEY` и `USE_IN_MEMORY_DATA=false`

## 3. Автоматические тесты

### 3.1. Unit-тесты

Файл:

- [geo.test.ts](../backend/tests/geo.test.ts)

Проверяют:

- нулевое расстояние для одинаковых координат;
- зависимость количества очков от точности ответа;
- корректное отображение `accuracyLabel` по диапазонам расстояния.

### 3.2. Интеграционные тесты API

Файл:

- [api.test.ts](../backend/tests/api.test.ts)

Проверяют:

- `GET /api/health`
- `GET /api/themes`
- валидацию некорректного запроса на создание сессии
- `POST /api/question-suggestions`
- прохождение полной 10-вопросной сессии
- обновление лидерборда и ранга после завершения игры

Важно:

- эти тесты используют `MemoryQuizRepository`
- прогон тестов не зависит от внешней БД и не требует `Supabase`

### 3.3. CI на GitHub

Файл:

- [ci.yml](../.github/workflows/ci.yml)

На каждом `push` и `pull_request` выполняются:

- установка зависимостей backend;
- сборка backend;
- запуск backend-тестов;
- установка зависимостей frontend;
- сборка frontend.

## 4. Результаты автоматического прогона

Последний локальный прогон автора перед обновлением документации:

- `backend build` - `Passed`
- `backend tests` - `8 passed`
- `frontend build` - `Passed`

Команды, которыми это проверялось:

```bash
docker run --rm -v "${PWD}:/src" -w /tmp node:20-alpine sh -lc "cp -r /src/backend ./backend && cd backend && npm ci && npm run build && npm test"
docker run --rm -v "${PWD}:/src" -w /tmp node:20-alpine sh -lc "cp -r /src/frontend ./frontend && cd frontend && npm ci && npm run build"
```

## 5. Ручные smoke-тесты

Ниже приведены сценарии, которые проверяют уже не только backend-контракты, но и реальную пользовательскую работу с приложением.

Этот сценарий соответствует публичной сдаче проекта:

- проект скачивается из GitHub;
- выполняется `docker compose up --build`;
- приложение запускается без ключей и без внешней БД;
- backend использует `in-memory` режим по умолчанию.

## 7. Итог

Сейчас у проекта есть два уровня проверки:

- реальные автотесты, которые гоняются локально и в CI;
- ручные smoke-тесты, подтверждающие, что приложение запускается и проходит основной пользовательский сценарий.

Для тестового fullstack-проекта это даёт хороший баланс между инженерной проверкой API и демонстрацией реальной работоспособности интерфейса.
